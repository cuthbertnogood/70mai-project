# План аналитики по логам автопилота 70mai (обе SD-карты / периоды)

**Дата создания:** 2026-07-16  
**Проект:** /Users/cuthbert/work_local/70mai_project  
**Цель:** Найти подходы (параллелизм vs последовательность, размеры чанков, профили и т.д.), которые давали **максимальную скорость потока** на каждом шаге обработки видео.

## Шаги анализа (что измеряем)

1. **Копирование с SD** — перенос 1-минутных клипов с карты на SSD.
2. **Merge (склейка)** — объединение 1-минутных клипов в длинные файлы NO_/PA_/EV_*.mp4 (ffmpeg concat -c copy).
3. **Compose** — объединение Front + Back в одно stacked-видео (2-камерное).
4. **Загрузка на YouTube** — отправка готовых файлов.

Для каждого шага фиксируем:
- Реализованный подход (параллелизм/sequential, batch size, параметры).
- Объёмы (GB, минуты видео, кол-во клипов/трипов).
- Время выполнения.
- Скорость (MB/s, GB/час, realtime factor, clips/час).
- Победные конфигурации.

## 2. Общий метод анализа

1. Построчно парсим лог (regex + парсинг timestamp).
2. Группируем события по **run** (по времени "publish_all start" / "import_70mai start" + типы).
3. Для каждого run извлекаем:
   - Полную командную строку (все флаги).
   - Объёмы данных.
   - Времена начала/окончания каждой фазы.
   - Вычисляем throughput.
4. Строим таблицы "конфиг → throughput".
5. Выделяем лучшие конфигурации по каждому шагу и по pipeline в целом.
6. Учитываем системные факторы (SSD, Amphetamine, свободное место, время суток).

**Ключевые метрики throughput:**
- Копирование: MB/s, время на сессию.
- Merge: MB/s effective, минуты видео в час, clips/min.
- Compose: realtime factor (speed X.XXx), время на 1 час 2-камерного видео, GB/час вывода.
- Upload: MB/s (peak + avg), время на GB.
- Общий: сколько реального времени видео проходит весь путь за час работы.

## 3. Детальный разбор по шагам

### Шаг 1: Копирование с SD

**Индикаторы в логах:**
- "copying clip", "clip copied"
- "Found X clips in ..."
- "Probing clip durations"
- "Probe: 8 parallel ffprobe workers"
- Строки про сессии перед merge

**Реализация (из кода):**
- `lib/import_70mai.py`
- `PROBE_WORKERS = 8` (ThreadPoolExecutor в `plan_estimate.py` и import).
- Копирование само по себе — преимущественно последовательное (shutil + логика по клипам).
- Есть отдельный "copy conveyor" для статуса.

**Что анализировать:**
- Время копирования всех клипов одной сессии/типа.
- Оверхед копирования (copy vs merge).
- Влияние параллельного probing (8 воркеров).
- Различия между апрельскими и июльскими запусками.

**Вопросы для максимума:**
- Всегда ли probing на 8 потоках был оптимальным?
- Когда copy был bottleneck'ом (медленная SD, много мелких файлов)?

### Шаг 2: Merge (склейка 1-мин клипов)

**Индикаторы в логах:**
- "Merging ...", "merging 10 clips → NO_...mp4"
- "=== Merging ... ==="
- "Done in Xm: N merged, Y skipped, Z failed"
- В старых логах явно: `ffmpeg concat -c copy (lossless, no re-encode)`

**Ключевые параметры (код + логи):**
- `MERGE_WORKERS = 1` (жёсткое решение: комментарий в коде — "USB/SD seeks worse with parallel concat").
- `--chunk-minutes 10.0` (в импорте).
- `MERGE_DURATION_TOLERANCE = 0.85` (Normal), `0.98` (Event/Parking).
- `MERGE_HEARTBEAT_SEC`, retry-логика.
- `PROBE_WORKERS = 8` (пробинг параллельный, merge — нет).

**Примеры из логов:**
- "Done in 38m 39s: 2 merged..."
- "Done in 1h 38m 13s: 70 merged..."
- Старые: "== Merging output files: 920/920 ..."

**Что анализировать:**
- Время merge на X GB / Y минут видео.
- Эффективная скорость (MB/s).
- Влияние размера чанка (10 мин).
- Разница Normal vs Event/Parking (разные tolerances и размер файлов).

**Гипотеза (нужно подтвердить цифрами):**
Максимум почти всегда достигается при `MERGE_WORKERS=1` + последовательной обработке по сессиям. Параллельный merge вредит из-за случайных доступов на SD/USB.

### Шаг 3: Compose (объединение Front + Back)

**Индикаторы в логах:**
- "=== Chunk X [Type] ..."
- "run_compose_2cam"
- "Encode: [█████████████████████░░░░░░░░░░░░░░░] ... speed 2.07x"
- "trip_XX.mp4", ETA, elapsed.

**Ключевые параметры:**
- `--profile` (balanced | draft | quality | hevc).
- HW acceleration (videotoolbox, scale_vt, несколько fallback'ов).
- Размер чанка в паблише: `--chunk-minutes 120` (target ~120 мин).
- Per-trip vs batched compose.

**Наблюдения из логов (июль 2026):**
- Скорости от ~0.47x до 2.07x.
- На длинных кодированиях стабильно ~2.0–2.07x.

**Что анализировать:**
- Максимальный realtime factor по профилям.
- Время на час 2-камерного видео.
- Зависимость от длины чанка и типа контента.
- Сравнение с апрельскими запусками.

**Параллелизм:** В основном sequential (per trip / per chunk). Стоит проверить, можно ли безопасно запускать несколько compose параллельно.

### Шаг 4: Загрузка на YouTube

**Индикаторы:**
- "Uploading to YouTube: ... trip_XX.mp4"
- Прогресс: "XX GB (YY%) | 15.7 MB/s | ETA ..."
- "Uploaded: https://youtu.be/..."
- `youtube_upload.diag.jsonl`: `upload_start`, `chunk_ok`, `upload_success`, `throughput_mbps`, `avg_throughput_mbps`, `elapsed_sec`.

**Ключевые параметры:**
- `--per-trip-upload` (или загрузка по чанкам 120 мин).
- Размер chunk'а загрузки (пример: 64 МБ).
- Resume (`resume: True` в многих случаях).
- `--resume-upload`.

**Наблюдения:**
- Скорость сильно плавает: пики до ~19.9 MB/s, часто 2–8 MB/s, иногда <1 MB/s.
- Много resume-загрузок.

**Что анализировать:**
- Пиковый и средний MB/s.
- Время на файл разного размера.
- Влияние размера (мелкие трипы vs большие чанки).
- Оверхед resume и сетевые условия.
- Лучшие периоды (время суток?).

## 4. Как находить точку максимальной производительности

1. **Парсинг и таблицы** (pandas + regex):
   - Извлечь все run'ы.
   - Для каждой фазы — duration + volume → throughput.
   - Присоединить флаги запуска.

2. **Группировки для сравнения:**
   - По периоду (апрель vs июль).
   - По типу (Normal / Event / Parking).
   - По значениям `--chunk-minutes`, `--profile`, `merge_workers`.
   - По `--per-trip-upload` vs batched.

3. **Критерии победы:**
   - Самый высокий throughput для шага.
   - Лучший общий pipeline (сколько видео в час проходит от SD до YT).
   - Стабильность (меньше variance).

4. **Учёт ограничений:**
   - SD/USB seek performance → merge sequential.
   - CPU/GPU для compose.
   - YouTube quota + сеть для upload.
   - Дисковый ввод-вывод (SSD staging).

5. **Дополнительные разрезы:**
   - Зависимость от количества свободного места.
   - Влияние Amphetamine / предотвращения сна.
   - Ошибки и stall'ы как маркеры плохих конфигураций.

## 5. Ключевые конфигурации (из кода и логов)

- **Probing:** `PROBE_WORKERS = 8` (почти всегда параллельно).
- **Merge:** `MERGE_WORKERS = 1` (по дизайну, с комментарием про SD seeks). Поддерживается `--merge-workers`.
- **Chunking:** import — `--chunk-minutes 10`, publish/compose — `--chunk-minutes 120`.
- **Compose:** разные профили (`balanced` по умолчанию), HW acceleration (videotoolbox).
- **Upload:** resumable, chunked, часто с `--per-trip-upload`.
- **Другие флаги:** `--gap-seconds 120.0`, `--state-on-sd`, `--continue-on-error`, `--resume`.
